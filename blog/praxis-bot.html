How I Built a RAG Research Bot for My Neuroscience Lab

The PRAXIS Bot indexes 500+ neuroscience papers and answers protocol questions in Slack. Here's the architecture, the trade-offs, and what I'd do differently.

The Problem

Our lab had a knowledge bottleneck. Eight people working across fMRI preprocessing, computational modeling, and clinical data analysis, all asking variations of the same questions: "What's the motion threshold?" "Which normalization template are we using?" "What did Power et al. say about scrubbing?"

The answers existed, scattered across PDFs, Google Docs, and tribal knowledge in people's heads. Finding them took 20-30 minutes per question. Multiply that across eight people, and we were burning hours every week on information retrieval instead of research.

The Architecture

I built a retrieval-augmented generation (RAG) system deployed as a Slack bot. The pipeline is straightforward:

Ingestion PyMuPDF + custom chunker for PDFs, protocols, and lab docs
Embedding sentence-transformers (all-MiniLM-L6-v2) for semantic search
Vector Store FAISS with IVF-PQ indexing for sub-second retrieval
Generation LangChain + Ollama (local LLM) for answer synthesis

The key design decision was running the LLM locally. We handle sensitive research data and pre-publication manuscripts, so sending queries to OpenAI was off the table. We use Qwen3:14b via Ollama, which runs comfortably on our lab Mac with 32GB RAM.

The Chunking Strategy

Getting chunking right was the hardest part. Academic papers have structure that generic chunkers destroy. My approach:

  • Section-aware splitting — chunks respect paper section boundaries (Methods stays with Methods)
  • Overlap of 128 tokens — enough context to avoid orphaned references
  • Metadata injection — each chunk carries the paper title, authors, year, and section header as prefix metadata
  • Protocol priority — lab-specific protocols get 2x weight in the retrieval ranking

This metadata injection was critical. Without it, the bot would return technically correct answers from the wrong paper. With it, answers include proper citations, which our team trusts more.

Deployment: Slack Socket Mode

The bot runs on Slack's Socket Mode, which means no public endpoint. It connects outbound to Slack's WebSocket, so it works behind our university firewall with zero port forwarding.

# Core event loop (simplified)
@app.event("app_mention")
def handle_mention(event, say):
    query = event["text"]
    docs = retriever.get_relevant_documents(query)
    answer = chain.invoke({
        "question": query,
        "context": format_docs(docs)
    })
    say(answer, thread_ts=event["ts"])

I run it as a launchd daemon on macOS, so it auto-starts on boot and restarts on crash. Total uptime since deployment: 99.7%.

Results

After three months in production:

  • 500+ papers indexed across neuroscience, psychology, and computational methods
  • Literature search dropped from 30 min to under 60 seconds for common questions
  • 8 active users across the lab team
  • Zero data leakage incidents — everything runs local, nothing leaves the network

What I'd Do Differently

If I rebuilt it today, I'd use ColBERT v2 instead of dense embeddings + FAISS. Late-interaction retrieval handles the long-tail queries (specific figure references, statistical thresholds) that dense embeddings miss. I'd also add a feedback loop so team members can thumbs-up/down answers, creating a continuous improvement signal without manual curation.
